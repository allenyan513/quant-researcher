# quant-researcher — Features(需求讨论稿 v0.1)

> 状态:讨论中 · 仅描述**需求**,不含实现方案 · 2026-05-19

## 1. 背景与目标

我已有一套围绕美股的个人投研工作流,但能力分散在不同地方:`quant-engine`(回测)、`valuescope`(估值 SaaS)、若干 Claude skills(盘前/盘后/深度调研/周报)、以及一个 FMP MCP。它们各自为政,Claude Code 无法把"一句自然语言"顺畅地拆解成跨能力的完整研究链路。

**quant-researcher 的目标**:做一个**给 Claude Code 用的辅助投研底座**。Claude Code 是唯一主入口,负责理解自然语言、拆解任务、按需调用本系统的**可组合数据/分析能力**,再自己合成结论与叙述。本系统只负责"取数 + 计算 + 持久化 + 可复现",不负责"写文章"——叙述交给 Claude。

定位:**个人工作流优先**。围绕我自己的持仓/IBKR/Notion 习惯做强约定,少配置;开源但不为通用性牺牲顺手程度。

## 2. 设计原则(需求层面的约束)

1. **Claude 是编排者,本系统是底座**:对外暴露**粒度小、可组合**的操作(筛选 / 估值 / 取财报 / 算信号 / 记决策 / 跑回测),而不是一个"端到端黑箱"。Claude 负责把它们串起来。
2. **稳定的机读契约**:每个操作返回**结构化、带 schema、带 as-of 时间戳**的结果,Claude 能可靠地链式消费(筛选 → 选标的 → 估值 → 深研 → 模拟买入 → 跟踪)。
3. **宇宙数据仓库是脊柱**:筛选/技术扫描/估值/信号/回测全部依赖一个本地的、可离线查询的美股宇宙数据仓库(我已确认走这条路,接受数据管线维护成本)。
4. **一切可复现、可审计**:任何筛选/估值/研报/决策/信号/回测都要落盘**输入快照 + 参数 + 代码或模型版本**。让"放手让 Claude 决策,事后复盘"这件事可信。
5. **point-in-time 正确**:筛选与回测必须能按"某历史时点的已知信息"取数,避免前视偏差;退市标的保留(抗幸存者偏差)。
6. **新鲜度与成本透明**:每个结果都告诉 Claude"数据多新、是否走了昂贵刷新",Claude 据此决定是否触发刷新。
7. **个人约定优先**:持仓来源沿用现有 IBKR/CSV 机制;呈现/分发沿用现有 skills 与 Notion。本系统提供数据,现有 skill 负责排版输出——边界要清晰。

## 3. 核心概念

- **Universe(宇宙)**:纳入研究的美股集合(普通股为主)。需定义:覆盖范围、纳入/剔除规则(ADR/OTC/SPAC/最小流动性)、是否含退市标的。
- **Warehouse(数据仓库)**:本地持久化的宇宙数据(档案、报表、估值快照、价格历史、衍生指标),分层刷新。
- **Operation(原子操作)**:Claude 可单独调用的最小能力单元,输入/输出有 schema。
- **Workflow(组合流程)**:少量高层流程(如"深度投研"="取数+估值+技术+持股结构+财报"的编排),本质是 Operation 的有序组合。
- **Snapshot(快照)**:某操作执行时所见数据的不可变记录,用于复现与历史对比。
- **Decision Ledger(决策账本)**:Claude 每一笔模拟买卖决策 + 理由 + 当时数据快照 + 远期收益跟踪。

## 4. 能力域

> 9 个原始场景已被细化、扩充,并按依赖关系重组为 8 个能力域。A 是其余一切的基础。

### A. 宇宙数据仓库(基础设施 · 服务全部场景)

- 从 FMP 批量入库美股宇宙(参考 valuescope 的 `updateTicker()`+ 薄编排 + 分层刷新模式)。
- **分层刷新**:价格/成交量(日)、报表/盈利(随财报事件)、公司档案(月)。降低 FMP 配额消耗。
- **存储内容**:公司档案、市值/估值倍数、三大报表(含 TTM)、关键比率与增长、价格历史 OHLCV、衍生技术指标、估值快照、盈利日历、内部人/13F/分析师一致预期。
- **point-in-time**:记录每条数据的"知悉时间",支持"按 T 日已知信息"查询。
- **数据质量**:缺失/异常打标并暴露给 Claude;每次查询返回 as-of 与覆盖率。
- **新鲜度策略**:Claude 可查"某标的/某字段多新",并可显式触发"针对性刷新"(单票/单板块),避免每次全量。
- *(讨论点见 §7:这一层与 valuescope 高度重叠——是建模重写、复用代码,还是直接调 valuescope。)*

### B. 选股筛选(场景 #1 #2 · 统一筛选引擎)

把"基本面筛选"和"技术信号扫描"统一成**一个可组合筛选引擎**,因为真实需求常是二者交集。

- **基本面条件**:支持**衍生与比较式条件**——不止 `marketCap > 10B`,还要 `forwardPE < PE`、`PEG < 1`、`FCF yield > x`、行业相对分位等。这是 FMP 自带筛选器做不到、本项目必须自建的核心。
- **技术信号扫描**(全市场):基于价格历史计算并扫描——MACD 金叉/死叉、均线交叉、RSI 区间、52 周新高/低、放量突破、形态。需要"近 N 日内触发"的时间窗与新鲜度截断。
- **组合条件**:`MACD 金叉 AND 市值>10B AND PE<30` 一次表达。
- **命名筛选定义**:可保存复用的 screen(如"我的 GARP""深度价值"),可重跑。
- **结果可持久化、可 diff**:支持"自上次以来新进/移出我筛选池的标的"——直接喂给早报/决策账本。
- **输出**:可排序/可排名、稳定 schema、带 as-of。

### C. 估值引擎(场景 #3)

- **多模型**:DCF/FCFF(增长退出 & EBITDA 退出)、反向 DCF、P/E 与 EV/EBITDA 倍数、PEG、EPV、DDM、营收 DCF、分析师一致预期(参考 valuescope 已实现的 9 模型)。
- **假设显式且可调**:增长率/WACC/利润率/退出倍数等可被 Claude 或我覆盖;返回**敏感性矩阵**与同业对比。
- **输出**:合理价值区间、上行/下行空间、低/高估判定、关键假设、同业可比。
- **估值快照**:落盘可追踪、可历史对比、可被深研/早报复用;存输入+模型版本以便复现。

### D. 深度投研 + 财报速读(场景 #4 #6)

- **深度投研**:聚合一切——基本面趋势、估值(复用 C)、技术面、新闻、内部人/13F、分析师、盈利电话纪要——产出**结构化数据包**:业务与护城河、财务趋势、估值、风险、资本配置、持股结构、催化剂、多空论点。
  - **分工**:本系统出"数据包 + 量化结论";Claude / 现有 `stock-deep-dive` skill 负责合成 Morningstar 式叙述与排版(HTML dashboard)。
  - 研报与其底层数据快照一并落盘 → 可复现、可与未来对比。
- **财报速读**(#6):最新财报 actual vs estimate、纪要要点、分部趋势、指引、利润率/现金趋势、资产负债健康;**自动对比上一版估值/论点**(论点是否被打破),并可触发估值刷新(C)与早报(E)。

### E. 持仓早报 / Morning Call(场景 #5)

- **持仓来源**:沿用现有 IBKR Flex / CSV 机制(个人约定,不重造)。
- **逐持仓**:隔夜涨跌、相关新闻、技术信号(复用 B)、估值状态(复用 C)、临近财报、分析师变动。
- **组合层**:暴露/集中度/因子倾斜、较昨日有何变化。
- **联动决策账本**:标注哪些持仓是 Claude 决策仓位、跟踪表现如何(复用 F)。
- **输出**:数据交给现有早报 skill / Notion(个人约定),本系统只供数据。

### F. 决策账本 / 模拟盘(场景 #7 · 形态=决策日志+收益跟踪)

> 不做完整模拟组合;做**Claude 判断力的可复盘战绩**。

- **记录每笔决策**:标的、动作(买/卖/减/加/回避)、日期、决策时价格、**论点文本**、信心度、触发来源(哪次研究/筛选/信号引发)、Claude 模型与版本。
- **决策时数据快照**(不可变,参考 valuescope 的 immutable critique 模式)→ 事后能还原"Claude 当时看到了什么"。
- **远期收益跟踪**:1 周 / 1 / 3 / 6 月,对比 SPY 与所属行业 ETF(算超额 alpha)。
- **决策计分卡**:命中率、平均 alpha、按论点类型/行业/信心度分组——让 Claude 自我复盘、让我审计 Claude 的判断质量。
- **状态机**:open/closed、论点是否仍成立、复盘触发(财报/到目标价/止损位)。
- 明确区别于真实组合——这是**判断力 track record**,轻量。

### G. 量化信号研究(场景 #8)

- **信号研究流程**:定义假设 → 在宇宙+历史上计算因子/指标 → 评估预测力(IC、分位收益、衰减)→ 产出信号序列。
- 复用 `quant-engine` 的指标/分析能力;宇宙历史由仓库(A)提供。
- **信号可持久化、可版本化**,并能反哺筛选(B)、回测(H)、决策账本(F)。
- 研究规格(假设+参数)与结果一并落盘,可复现。

### H. 策略回测(场景 #9)

- Claude 编写策略规格 → 在回测引擎运行(`quant-engine` 已有,建模/复用)。
- 仓库提供**抗幸存者偏差**的宇宙与历史;point-in-time 取数。
- **输出**:标准指标(Sharpe/Sortino/Alpha/最大回撤)、成交日志、净值曲线;回测运行落盘可横向对比。

### 贯穿主线(F↔G↔H)

一条统一闭环:**回测验证的策略(H)→ 上线产出实时信号(G)→ 触发 Claude 模拟决策(F)→ 远期战绩反过来评估策略与 Claude 判断**。这条线让 9 个场景不是孤岛。

## 5. 贯穿性需求(横切关注点)

- **可复现 & 审计**:每个产出存 输入快照 + 参数 + 代码/模型版本。这是"放手让 Claude 决策"可信的前提。
- **机读契约**:所有操作输出稳定 JSON schema + as-of;Claude 可无歧义链式调用。
- **自然语言任务拆解契约**:本系统提供**细粒度可组合原语** + 少量高层 workflow,把"如何拆解"的智能留给 Claude,不在系统内做 NLP。
- **持久化 & 历史 diff**:筛选/估值/研报/决策/信号/回测全部入库 → 支持"较上次有何变化""战绩追踪"。
- **新鲜度 & 成本透明**:每个结果自带数据新鲜度与是否触发昂贵刷新。
- **轻量"自上次以来有何新"**:v1 不做推送基础设施,但要可查询增量(喂早报/决策账本)。
- **Watchlist / 命名宇宙**:个人清单可喂入筛选/早报/信号。
- **个人输出约定**:Notion / 现有 skills / IBKR 持仓源为既定约定,系统供数、skill 呈现,边界清晰。

## 6. 非目标 / v1 边界

- ❌ 真实券商下单/实盘执行(纯研究与模拟)。
- ❌ 实时 tick / 分钟级 / 盘中数据;❌ 期权 / 期货(与 quant-engine 现有边界一致,日频为主)。
- ❌ 多用户 / 鉴权 / SaaS 化(个人优先)。
- ❌ 本系统内做自然语言理解(那是 Claude Code 的职责)。
- ❌ 本系统写"文章/叙述"(交给 Claude 与现有 skills)。
- ⏳ 实现方案(MCP server vs CLI vs 库 vs REST)、技术栈、DB 选型——**本稿不决定**,定稿后单独规划。

## 7. 开放讨论项(需要一起拍板)

1. **与 valuescope 的关系(最大问题)**:A(数据仓库)+ C(估值)+ E(早报)与 valuescope 高度重叠,且它已自带 MCP server。三选一:**(a)** 参考其模式在本仓库干净重写(个人优先、不依赖);**(b)** 复用其代码;**(c)** 本系统编排 + 直接调 valuescope 的 MCP。各自工作量与维护负担差异巨大。
2. **与 quant-engine 的关系**:G/H 直接复用其引擎与指标,还是重建?是否把它作为依赖/子模块?
3. **宇宙范围与配额**:全 ~8000 标的 vs 先聚焦个人关注 + 主要指数成分?FMP 套餐配额能否支撑全宇宙日度刷新?
4. **point-in-time 严格度**:历史报表/估值要做到多严格的时点正确?(影响仓库复杂度)
5. **决策账本基准**:除 SPY + 行业 ETF,是否要因子/同风格基准?
6. **v1 内部排序**:9 场景虽全要,但 A 必须先行;建议顺序 A → B → C → (D/E/F) → G → H,是否认可?
7. **开源边界**:个人约定(IBKR/Notion)如何在开源版里脱敏/可替换?
